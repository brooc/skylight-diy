# v0.1 Implementation Notes

This document captures small implementation details that should be handled during v0.1 but do not need to expand the already-detailed v0.1 implementation plan.

## Database migration notes

### Enable UUID generation

The schema uses `gen_random_uuid()` for primary keys. The first migration should enable the required Postgres extension:

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;
```

Acceptance criteria:

- [ ] First migration enables `pgcrypto`.
- [ ] Tables using `gen_random_uuid()` can be created on a fresh Postgres database.
- [ ] Seed script works against a clean database.

## Session and OAuth cookie notes

The app uses HTTP-only cookies for local admin unlock state and Google OAuth state validation.

Cookie defaults for v0.1:

- [ ] `HttpOnly: true`
- [ ] `SameSite: Lax`
- [ ] `Path: /`
- [ ] `Secure: false` in local HTTP development
- [ ] `Secure: true` when deployed behind HTTPS

Why `SameSite=Lax`:

- Google OAuth returns through a top-level redirect.
- `Lax` is usually strict enough for local app cookies while still allowing the OAuth callback flow to work.

## Token encryption key generation

`TOKEN_ENCRYPTION_KEY` must be a base64-encoded 32-byte key.

Generate a local development key with:

```bash
openssl rand -base64 32
```

Rules:

- [ ] Put the generated value in local `.env`.
- [ ] Do not commit `.env`.
- [ ] Back up the key with other deployment secrets.
- [ ] Losing the key means existing encrypted OAuth tokens cannot be decrypted.
- [ ] If the key is lost, the user can reconnect Google Calendar after restoring the app database.

## Backup and restore notes

The basic v0.1 restore command assumes the target database is empty or intentionally being overwritten.

Before documenting restore as a user-facing command, include a warning:

> Restoring into a non-empty database can duplicate or conflict with existing data. For v0.1, restore into a fresh database unless you know exactly what you are doing.

Acceptance criteria:

- [ ] README or setup docs explain what data lives in Postgres.
- [ ] README or setup docs explain that `.env` and `TOKEN_ENCRYPTION_KEY` must be backed up separately.
- [ ] Restore instructions warn about restoring into a non-empty database.

## First-run setup details

v0.1 should avoid ambiguous startup behavior.

Rules:

- [ ] If no household exists, `/` redirects to `/setup`.
- [ ] If setup is complete, `/` redirects to `/today`.
- [ ] Setup completion should be idempotency-safe; a second setup attempt should not create a second household.
- [ ] v0.1 supports one household per deployment.
- [ ] Future multi-household support requires a separate ADR.

## Calendar cache details

The calendar display cache is replaceable display data.

Rules:

- [ ] Cache key includes household ID, range start, range end, timezone, and enabled source fingerprint.
- [ ] Source fingerprint changes when calendar source enabled state, display preferences, or person mapping changes.
- [ ] Fresh cache can be returned without calling Google.
- [ ] Expired cache should be refreshed from Google when possible.
- [ ] Stale cache may be returned with a warning when Google is unavailable.
- [ ] Stale cache should not be silently presented as fresh.
- [ ] Raw full Google event payloads should not be stored by default.

## Fire Tablet testing notes

The Fire Tablet is not just a demo target; it is the first real target device.

v0.1 manual test notes should capture:

- [ ] Fire Tablet model and Fire OS version.
- [ ] Browser tested, such as Silk Browser.
- [ ] Whether PWA install or Add to Home Screen was tested.
- [ ] Whether landscape mode is stable.
- [ ] Whether touch targets feel reliable.
- [ ] Whether the app survives a 2-4 hour soak test.
- [ ] Heat and charging observations.
- [ ] Any kiosk-browser experiments.
