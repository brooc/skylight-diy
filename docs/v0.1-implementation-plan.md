# v0.1 Implementation Plan

This document is the tactical build plan for v0.1. It should be detailed enough that a contributor can start from an empty checkout and implement the first real version of the app without needing additional architectural context.

v0.1 is not a throwaway prototype. It is the first narrow vertical slice of the real application.

## v0.1 goal

Build a self-hostable family dashboard that runs locally, uses Postgres, serves a tablet-friendly web UI, persists household data, and includes a read-only Google Calendar integration spike.

By the end of v0.1, a developer should be able to:

- [ ] Clone the repo.
- [ ] Run one setup command.
- [ ] Start Postgres, API, and web app locally.
- [ ] Open the dashboard in a browser or Fire Tablet.
- [ ] See seeded household members.
- [ ] See seeded chores and rewards.
- [ ] Mark a chore complete and see points update.
- [ ] See seeded meals for the week.
- [ ] Connect one Google account with read-only calendar access.
- [ ] Select calendars to display.
- [ ] Fetch Google Calendar events directly from Google for the displayed date range.
- [ ] Render Google events in Today and Week views.

## Explicit v0.1 non-goals

- [ ] No production user registration.
- [ ] No multi-household hosted SaaS model.
- [ ] No native mobile app.
- [ ] No calendar write support.
- [ ] No editing Google Calendar events.
- [ ] No canonical persistence of Google Calendar events.
- [ ] No recurring-event engine inside Skylight DIY.
- [ ] No Magic Import implementation beyond a placeholder screen.
- [ ] No PDF, image, OCR, or email ingestion.
- [ ] No offline-first sync.
- [ ] No payment or subscription infrastructure.

## Stack decisions for v0.1

These are implementation defaults for v0.1. Future ADRs can revise them, but v0.1 should not leave these choices open.

### Monorepo and tooling

- [ ] Use `pnpm` as the package manager.
- [ ] Use `pnpm workspaces` for monorepo package management.
- [ ] Use `Turborepo` for running dev/build/lint tasks across apps and packages.
- [ ] Use TypeScript everywhere.
- [ ] Use ESLint and Prettier.

Rationale:

- `pnpm` workspaces are simple and fast.
- Turborepo is optional but useful once there are multiple apps/packages.
- TypeScript keeps frontend, backend, database, and shared models aligned.

### Frontend

- [ ] Use Vite.
- [ ] Use React.
- [ ] Use TypeScript.
- [ ] Use Tailwind CSS.
- [ ] Use React Router.
- [ ] Use TanStack Query for API fetching and cache state.
- [ ] Add a PWA manifest, even if full offline support waits until later.

Rationale:

- The app is a browser/tablet dashboard, not a public SEO site.
- Vite keeps the frontend lightweight and fast.
- TanStack Query provides clean loading/error/refetch semantics for calendar reads.

### Backend

- [ ] Use Node.js.
- [ ] Use TypeScript.
- [ ] Use Fastify.
- [ ] Use REST endpoints.
- [ ] Use Zod for request/response validation.
- [ ] Use cookie-based local admin session for v0.1.

Rationale:

- Fastify is mature, fast, and works well for a modular API.
- REST is straightforward for the current scope.
- Zod gives runtime safety at API boundaries.

### Database

- [ ] Use Postgres.
- [ ] Use Drizzle ORM.
- [ ] Use Drizzle migrations.
- [ ] Use a Dockerized local Postgres instance.
- [ ] Use seed data for local development.

Rationale:

- Postgres is the target real database from day one.
- Drizzle keeps us close to SQL and migration files.
- We avoid SQLite/Postgres migration divergence.

### Calendar integration

- [ ] Use Google Calendar API.
- [ ] Use read-only calendar scope in v0.1.
- [ ] Store connected account metadata and encrypted tokens.
- [ ] Store selected calendar sources and display preferences.
- [ ] Fetch events directly from Google for the requested date range.
- [ ] Map Google events into display-only view models.
- [ ] Do not persist Google events as canonical app data.

## Local development commands

The final v0.1 developer workflow should support these commands:

```bash
pnpm install
pnpm db:up
pnpm db:migrate
pnpm db:seed
pnpm dev
```

Expected behavior:

- [ ] `pnpm install` installs all workspace dependencies.
- [ ] `pnpm db:up` starts Postgres through Docker Compose.
- [ ] `pnpm db:migrate` applies migrations.
- [ ] `pnpm db:seed` inserts local demo data.
- [ ] `pnpm dev` starts API and web app.

Stretch goal:

```bash
pnpm setup
```

- [ ] `pnpm setup` runs install, database startup, migrations, and seed data.

## Database schema

Implement schema in `packages/db/src/schema.ts` using Drizzle.

### Naming conventions

- [ ] Use snake_case table and column names.
- [ ] Use `uuid` primary keys.
- [ ] Use `created_at` and `updated_at` timestamps on app-owned entities.
- [ ] Use `timestamptz` for timestamps.
- [ ] Use `date` for meal plan dates when time is irrelevant.
- [ ] Use explicit foreign keys.
- [ ] Prefer enums only when the values are truly stable.

### `households`

Purpose: top-level household configuration.

Columns:

- [ ] `id uuid primary key default gen_random_uuid()`
- [ ] `name text not null`
- [ ] `timezone text not null default 'America/Los_Angeles'`
- [ ] `created_at timestamptz not null default now()`
- [ ] `updated_at timestamptz not null default now()`

Notes:

- v0.1 can seed exactly one household.
- Timezone should be used for date range calculations and all-day event display.

### `people`

Purpose: household members shown in the UI.

Columns:

- [ ] `id uuid primary key default gen_random_uuid()`
- [ ] `household_id uuid not null references households(id) on delete cascade`
- [ ] `display_name text not null`
- [ ] `color text not null`
- [ ] `role text not null default 'child'`
- [ ] `sort_order integer not null default 0`
- [ ] `created_at timestamptz not null default now()`
- [ ] `updated_at timestamptz not null default now()`

Valid roles for v0.1:

- `adult`
- `child`

### `chores`

Purpose: chore definitions.

Columns:

- [ ] `id uuid primary key default gen_random_uuid()`
- [ ] `household_id uuid not null references households(id) on delete cascade`
- [ ] `assigned_person_id uuid references people(id) on delete set null`
- [ ] `title text not null`
- [ ] `description text`
- [ ] `points integer not null default 1`
- [ ] `frequency text not null default 'daily'`
- [ ] `active boolean not null default true`
- [ ] `sort_order integer not null default 0`
- [ ] `created_at timestamptz not null default now()`
- [ ] `updated_at timestamptz not null default now()`

Valid v0.1 frequencies:

- `daily`
- `weekly`
- `once`

Notes:

- v0.1 can use simple frequency values instead of full recurrence rules.
- Full recurrence can be introduced later if needed.

### `chore_completions`

Purpose: record completed chores.

Columns:

- [ ] `id uuid primary key default gen_random_uuid()`
- [ ] `household_id uuid not null references households(id) on delete cascade`
- [ ] `chore_id uuid not null references chores(id) on delete cascade`
- [ ] `person_id uuid references people(id) on delete set null`
- [ ] `completed_for_date date not null`
- [ ] `points_awarded integer not null`
- [ ] `completed_at timestamptz not null default now()`

Constraints:

- [ ] Unique index on `(chore_id, completed_for_date)` for v0.1.

Notes:

- This prevents duplicate completion for the same chore/date.
- Later, shared chores may need a different uniqueness rule.

### `reward_redemptions`

Purpose: track manually redeemed rewards.

Columns:

- [ ] `id uuid primary key default gen_random_uuid()`
- [ ] `household_id uuid not null references households(id) on delete cascade`
- [ ] `person_id uuid not null references people(id) on delete cascade`
- [ ] `title text not null`
- [ ] `points_spent integer not null`
- [ ] `redeemed_at timestamptz not null default now()`

Notes:

- v0.1 does not need a separate rewards catalog unless it is easy.
- Reward balance can be calculated as completed points minus redeemed points.

### `meals`

Purpose: reusable meal ideas.

Columns:

- [ ] `id uuid primary key default gen_random_uuid()`
- [ ] `household_id uuid not null references households(id) on delete cascade`
- [ ] `name text not null`
- [ ] `notes text`
- [ ] `created_at timestamptz not null default now()`
- [ ] `updated_at timestamptz not null default now()`

### `meal_plan_entries`

Purpose: assign meals to dates.

Columns:

- [ ] `id uuid primary key default gen_random_uuid()`
- [ ] `household_id uuid not null references households(id) on delete cascade`
- [ ] `meal_id uuid references meals(id) on delete set null`
- [ ] `planned_date date not null`
- [ ] `slot text not null default 'dinner'`
- [ ] `custom_title text`
- [ ] `notes text`
- [ ] `created_at timestamptz not null default now()`
- [ ] `updated_at timestamptz not null default now()`

Valid v0.1 slots:

- `breakfast`
- `lunch`
- `dinner`
- `snack`

Notes:

- `custom_title` allows a plan entry without a reusable meal.

### `connected_accounts`

Purpose: store external provider account connections.

Columns:

- [ ] `id uuid primary key default gen_random_uuid()`
- [ ] `household_id uuid not null references households(id) on delete cascade`
- [ ] `provider text not null`
- [ ] `provider_account_id text`
- [ ] `display_name text`
- [ ] `email text`
- [ ] `encrypted_access_token text`
- [ ] `encrypted_refresh_token text`
- [ ] `access_token_expires_at timestamptz`
- [ ] `scopes text[] not null default '{}'`
- [ ] `created_at timestamptz not null default now()`
- [ ] `updated_at timestamptz not null default now()`

Valid v0.1 provider:

- `google`

Notes:

- Tokens must be encrypted before storing.
- If Google does not return a refresh token on later auth attempts, preserve the existing refresh token.

### `calendar_sources`

Purpose: store selected external calendars and display preferences.

Columns:

- [ ] `id uuid primary key default gen_random_uuid()`
- [ ] `household_id uuid not null references households(id) on delete cascade`
- [ ] `connected_account_id uuid not null references connected_accounts(id) on delete cascade`
- [ ] `provider text not null`
- [ ] `external_calendar_id text not null`
- [ ] `display_name text not null`
- [ ] `color text`
- [ ] `person_id uuid references people(id) on delete set null`
- [ ] `enabled boolean not null default true`
- [ ] `sort_order integer not null default 0`
- [ ] `created_at timestamptz not null default now()`
- [ ] `updated_at timestamptz not null default now()`

Constraints:

- [ ] Unique index on `(connected_account_id, external_calendar_id)`.

Notes:

- This table stores preferences only.
- It does not imply event ownership or event sync.

### `calendar_fetch_logs`

Purpose: basic observability for calendar reads.

Columns:

- [ ] `id uuid primary key default gen_random_uuid()`
- [ ] `household_id uuid not null references households(id) on delete cascade`
- [ ] `calendar_source_id uuid references calendar_sources(id) on delete set null`
- [ ] `range_start timestamptz not null`
- [ ] `range_end timestamptz not null`
- [ ] `status text not null`
- [ ] `error_message text`
- [ ] `fetched_at timestamptz not null default now()`

Valid status values:

- `success`
- `error`

Notes:

- This is not required for rendering but will make debugging Google reads easier.

## API design

Base URL in local dev:

```text
http://localhost:3000/api
```

### Health

#### `GET /api/health`

Purpose: verify API is alive.

Response:

```json
{
  "ok": true,
  "service": "skylight-diy-api"
}
```

Acceptance criteria:

- [ ] Returns HTTP 200.
- [ ] Does not require auth in v0.1.

#### `GET /api/household/current`

Purpose: return the seeded/current household and people.

Response shape:

```ts
type HouseholdResponse = {
```

Acceptance criteria:

- [ ] Returns seeded household.
- [ ] Returns people sorted by `sort_order`.

### Chores

#### `GET /api/chores/today`

Purpose: return today's chores and completion state.

Query params:

- `date` optional ISO date, default household-local today.

Response shape:

```ts
type ChoresTodayResponse = {
  date: string;
  chores: Array<{
    id: string;
    title: string;
    description?: string;
    points: number;

    completedAt?: string;

  }>;

```

Acceptance criteria:

- [ ] Shows active daily chores for the requested date.
- [ ] Includes weekly chores only when appropriate for v0.1's simple rule.
- [ ] Includes completion state.
- [ ] Includes reward balances by person.

#### `POST /api/chores/:choreId/complete`

Purpose: mark a chore complete for a date.

Request body:

```ts
type CompleteChoreRequest = {
  date: string;
};
```

Response shape:

```ts
type CompleteChoreResponse = {
  completion: {
    id: string;
    choreId: string;
    completedForDate: string;
    pointsAwarded: number;
    completedAt: string;
  };
};
```

Acceptance criteria:

- [ ] Creates a completion if none exists for that chore/date.
- [ ] Is idempotent if the chore is already complete.
- [ ] Awards the chore's configured points.
- [ ] Returns updated completion details.

#### `DELETE /api/chores/:choreId/complete?date=YYYY-MM-DD`

Purpose: undo a chore completion.

Acceptance criteria:

- [ ] Deletes completion for that chore/date.
- [ ] Returns HTTP 204 or updated chore state.

### Calendar events

#### `GET /api/calendar/events`

Purpose: fetch display events for a date range directly from Google Calendar.

Query params:

- `start` required ISO datetime.
- `end` required ISO datetime.
- `timezone` optional, default household timezone.

Response shape:

```ts
type CalendarEventsResponse = {
  rangeStart: string;
  rangeEnd: string;
  timezone: string;
  events: DisplayCalendarEvent[];
  sources: Array<{
    id: string;
    displayName: string;
    color?: string;
    personId?: string;
  }>;
  fetchedAt: string;
  errors: Array<{
    sourceId: string;
    message: string;
  }>;
};

type DisplayCalendarEvent = {
  id: string;
  provider: "google";
  externalId: string;
  calendarSourceId: string;
  externalCalendarId: string;
  title: string;
  start: string;
  end: string;
  allDay: boolean;
  location?: string;
  descriptionPreview?: string;
  color?: string;
  personId?: string;
  sourceName: string;
  htmlLink?: string;
};
```

Acceptance criteria:

- [ ] Loads enabled calendar sources.
- [ ] Fetches Google events for each enabled calendar.
- [ ] Uses Google-expanded date range results rather than local recurrence logic.
- [ ] Maps Google events to `DisplayCalendarEvent`.
- [ ] Sorts events by start time.
- [ ] Returns partial results if one calendar fails but another succeeds.
- [ ] Records basic fetch logs.
- [ ] Does not save fetched Google events into an events table.

### OAuth routes

#### `GET /api/integrations/google/connect`

Purpose: start OAuth flow.

Behavior:

- [ ] Generate a random `state` value.
- [ ] Store `state` in a signed HTTP-only cookie or short-lived server-side session.
- [ ] Redirect to Google OAuth URL.
- [ ] Request read-only calendar scope.
- [ ] Include `access_type=offline`.
- [ ] Include `prompt=consent` in local/dev so a refresh token is returned.

Required scope:

```text
https://www.googleapis.com/auth/calendar.readonly
```

#### `GET /api/integrations/google/callback`

Purpose: complete OAuth flow.

Behavior:

- [ ] Validate `state`.
- [ ] Exchange authorization code for tokens.
- [ ] Fetch user profile/email if available from token response or an appropriate Google endpoint.
- [ ] Encrypt access token.
- [ ] Encrypt refresh token when present.
- [ ] Store or update `connected_accounts` row.
- [ ] Redirect to frontend settings page.

Acceptance criteria:

- [ ] A connected Google account appears in Settings.
- [ ] Tokens are not stored in plaintext.
- [ ] Refresh token is preserved if subsequent OAuth response omits it.

### Token refresh

Implement helper:

```ts
async function getValidGoogleAccessToken(accountId: string): Promise<string>
```

Behavior:

- [ ] Load connected account.
- [ ] Decrypt access token.
- [ ] If token is not expired, return it.
- [ ] If expired, decrypt refresh token.
- [ ] Call Google token endpoint to refresh.
- [ ] Encrypt and persist new access token and expiry.
- [ ] Return new access token.

Acceptance criteria:

- [ ] Calendar reads work after initial access token expiry.
- [ ] Refresh failures return a useful error to the UI.

## Calendar provider module

Create an internal provider interface:

```ts
export type CalendarProviderName = "google";

export type CalendarProviderSource = {
  externalCalendarId: string;
  displayName: string;
  color?: string;
  primary?: boolean;
};

export type DisplayCalendarEvent = {
  id: string;
  provider: CalendarProviderName;
  externalId: string;
  calendarSourceId: string;
  externalCalendarId: string;
  title: string;
  start: string;
  end: string;
  allDay: boolean;
  location?: string;
  descriptionPreview?: string;
  color?: string;
  personId?: string;
  sourceName: string;
  htmlLink?: string;
};

export interface CalendarProvider {
  listSources(accountId: string): Promise<CalendarProviderSource[]>;
  listEvents(input: {
    accountId: string;
    sourceId: string;
    externalCalendarId: string;
    rangeStart: string;
    rangeEnd: string;
    timezone: string;
  }): Promise<DisplayCalendarEvent[]>;
}
```

Implementation notes:

- [ ] `GoogleCalendarProvider` implements this interface.
- [ ] `listEvents` should call Google `events.list` with `singleEvents=true`.
- [ ] `orderBy=startTime` should be used when `singleEvents=true`.
- [ ] `timeMin` and `timeMax` should come from the requested date range.
- [ ] All-day events should map Google `start.date` and `end.date` correctly.
- [ ] Timed events should map Google `start.dateTime` and `end.dateTime`.
- [ ] Event title should default to `(No title)` if missing.
- [ ] Description preview should be stripped/truncated.
- [ ] Do not expose raw full descriptions in v0.1 unless needed.
