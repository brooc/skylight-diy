# v0.1 Implementation Plan

**Status**: In progress (~72% complete). See [PROGRESS.md](../PROGRESS.md) for detailed step-by-step tracking.

This document is the tactical build plan for v0.1. It should be detailed enough that a contributor can start from an empty checkout and implement the first real version of the app without needing additional architectural context.

v0.1 is not a throwaway prototype. It is the first narrow vertical slice of the real application.

## Review-driven updates included

This revision incorporates the hardening pass from the planning review:

- [ ] First-run setup wizard.
- [ ] Local admin PIN for settings and integrations.
- [ ] Postgres-backed calendar event display cache.
- [ ] Concrete OAuth token encryption specification.
- [ ] Degraded state behavior.
- [ ] E2E and stability testing.
- [ ] Fire Tablet hardening acceptance criteria.
- [ ] Backup/restore basics.

## v0.1 goal

Build a self-hostable family dashboard that runs locally, uses Postgres, serves a tablet-friendly web UI, persists household data, and includes a read-only Google Calendar integration spike.

By the end of v0.1, a developer should be able to:

- [ ] Clone the repo.
- [ ] Run one setup command.
- [ ] Start Postgres, API, and web app locally.
- [ ] Complete first-run setup.
- [ ] Unlock settings with a local admin PIN.
- [ ] Open the dashboard in a browser or Fire Tablet.
- [ ] See household members.
- [ ] See chores and rewards.
- [ ] Mark a chore complete and see points update.
- [ ] See meals for the week.
- [ ] Connect one Google account with read-only calendar access.
- [ ] Select calendars to display.
- [ ] Fetch Google Calendar events for the displayed date range.
- [ ] Render Google events in Today and Week views.
- [ ] Return cached/stale display events when Google is temporarily unavailable.

## Explicit v0.1 non-goals

- [ ] No production user registration.
- [ ] No multi-household hosted SaaS model.
- [ ] No native mobile app.
- [ ] No calendar write support.
- [ ] No editing Google Calendar events.
- [ ] No canonical persistence of Google Calendar events.
- [ ] No recurring-event engine inside Skylight DIY.
- [ ] No Magic Import implementation beyond a placeholder screen.
- [ ] No PDF, image, OCR, or email ingestion.
- [ ] No full offline-first sync.
- [ ] No payment or subscription infrastructure.
- [ ] No Redis dependency.
- [ ] No remote telemetry by default.

Implementation notes:

- [ ] The API should fail fast if required env vars are missing.
- [ ] `TOKEN_ENCRYPTION_KEY` must decode from base64 to exactly 32 bytes.
- [ ] Do not commit real `.env` files.
- [ ] Losing `TOKEN_ENCRYPTION_KEY` means existing encrypted OAuth tokens cannot be decrypted.

## First-run setup wizard

When the API starts and no household exists, the web app should redirect to `/setup`.

Setup collects:

- [ ] Household name.
- [ ] Household timezone.
- [ ] Adult/admin display name.
- [ ] Optional household members.
- [ ] Local admin PIN.
- [ ] Optional demo data preference.

Setup creates:

- [ ] One household.
- [ ] One adult/admin person.
- [ ] Optional additional people.
- [ ] Admin PIN hash.
- [ ] `setup_completed_at` timestamp.
- [ ] Optional seed/demo chores and meals.

Admin PIN requirements:

- [ ] Store only a hash, never the raw PIN.
- [ ] Prefer Argon2id if practical; bcrypt is acceptable for v0.1 if Argon2id adds too much friction.
- [ ] Use a minimum PIN length of 4 digits for local-only v0.1.
- [ ] Allow changing the PIN later through Settings.
- [ ] Require the PIN for Settings, Google OAuth, and calendar source management.
- [ ] Do not require the PIN for passive dashboard viewing.

## Local admin session

v0.1 is a local/self-hosted appliance, not a production SaaS auth system.

Session behavior:

- [ ] Unlock admin access by submitting the local PIN.
- [ ] Store unlock state in an HTTP-only cookie.
- [ ] Use a signed session cookie or server-side session.
- [ ] Expire admin unlock after an idle timeout.
- [ ] Allow manual lock from Settings.
- [ ] Protect integration routes with admin session middleware.

Protected routes:

- [ ] Google OAuth connect.
- [ ] Google OAuth callback finalization.
- [ ] Calendar source import.
- [ ] Calendar source preferences.
- [ ] Household settings.
- [ ] Admin PIN change.

Unprotected routes:

- [ ] Today dashboard read.
- [ ] Week calendar read.
- [ ] Chores view and completion.
- [ ] Meals view.

## Database schema

- [ ] Use snake_case table and column names.
- [ ] Use `uuid` primary keys.
- [ ] Use `created_at` and `updated_at` timestamps on app-owned entities.
- [ ] Use `timestamptz` for timestamps.
- [ ] Use `date` for meal plan dates when time is irrelevant.
- [ ] Use explicit foreign keys.
- [ ] Prefer stable text values over premature enum types.

## Calendar cache behavior

Cache key inputs:

- [ ] Household ID.
- [ ] Range start.
- [ ] Range end.
- [ ] Timezone.
- [ ] Enabled calendar source IDs.
- [ ] Calendar source updated timestamps or a source fingerprint.

Suggested TTLs:

- [ ] Fresh cache: 5 minutes.
- [ ] Stale-but-usable cache: 24 hours.

Read flow:

1. UI requests calendar events for a date range.
2. API computes cache key.
3. API returns fresh cache immediately if available.
4. If cache is missing or expired, API fetches from Google.
5. If Google succeeds, API maps events, overwrites cache, and returns fresh data.
6. If Google fails and stale cache exists, API returns stale data with degraded-state metadata.
7. If Google fails and no cache exists, API returns a friendly error state.

## Token encryption specification

Use AES-256-GCM with Node.js built-in `crypto`.

Key requirements:

- [ ] `TOKEN_ENCRYPTION_KEY` is base64 encoded.
- [ ] Decoded key must be exactly 32 bytes.
- [ ] API fails fast if key is missing or invalid.

Encryption requirements:

- [ ] Use a random 12-byte IV per encrypted value.
- [ ] Store ciphertext with GCM auth tag.
- [ ] Store encrypted values as:

```text
v1:<base64_iv>:<base64_ciphertext>:<base64_auth_tag>
```

Operational requirements:

- [ ] Never store OAuth tokens in plaintext.
- [ ] Never log plaintext tokens.
- [ ] Never log full OAuth token responses.
- [ ] Never send OAuth tokens to the frontend.
- [ ] Preserve an existing refresh token when Google omits a new one during reconnect.
- [ ] Mark account `reauthorization_required` if refresh fails permanently.

### Calendar events

- [ ] `GET /api/calendar/events?start=...&end=...&timezone=...`

Returns:

```ts
type CalendarEventsResponse = {
  rangeStart: string;
  rangeEnd: string;
  timezone: string;
  events: DisplayCalendarEvent[];
  sources: CalendarDisplaySource[];
  fetchedAt?: string;
  cacheStatus: "fresh" | "refreshed" | "stale" | "miss";
  degraded: boolean;
  warnings: Array<{
    code: string;

    sourceId?: string;

```

Acceptance criteria:

- [ ] Loads enabled calendar sources.
- [ ] Uses cache when fresh.
- [ ] Fetches Google events for each enabled source when cache is expired or missing.
- [ ] Uses Google-expanded date range results rather than local recurrence logic.
- [ ] Maps Google events to `DisplayCalendarEvent`.
- [ ] Sorts events by start time.
- [ ] Returns partial results if one calendar fails but another succeeds.
- [ ] Returns stale cache with warnings when Google fails and stale cache exists.
- [ ] Records fetch logs.
- [ ] Does not save fetched Google events into a canonical events table.

## Degraded state behavior

| Failure | v0.1 behavior |
| --- | --- |
| Google network failure | Show stale cached events if available, with "Last updated" warning. |
| Google API rate limit | Show stale cached events if available, with retry warning. |
| Refresh token expired or revoked | Show stale cached events if available, mark account as needing reconnect. |
| Calendar deleted externally | Show warning for that source and allow disabling/removing it. |
| One calendar fails | Show partial results from successful calendars. |
| No Google account connected | Show empty calendar state with Settings prompt. |
| Cache missing and Google fails | Show friendly error state, not a broken page. |

## Backup and restore basics

v0.1 must document enough to prevent obvious data loss.

Document:

- [ ] App-owned data lives in Postgres.
- [ ] Docker Compose uses a named Postgres volume.
- [ ] Users must back up the Postgres database or Docker volume.
- [ ] Users must separately back up `.env` and `TOKEN_ENCRYPTION_KEY`.
- [ ] Losing `TOKEN_ENCRYPTION_KEY` means stored Google tokens cannot be decrypted.
- [ ] If the key is lost, users can reconnect Google Calendar after restore.

Basic backup command:

```bash
docker exec skylight-diy-postgres pg_dump -U skylight skylight_diy > skylight_diy_backup.sql
```

Basic restore command:

```bash
cat skylight_diy_backup.sql | docker exec -i skylight-diy-postgres psql -U skylight skylight_diy
```

v0.2 should add a formal backup/restore guide.
